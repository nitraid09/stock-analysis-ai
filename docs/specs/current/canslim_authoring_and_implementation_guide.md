# CANSLIM分析基盤 authoring / implementation guide

## 1. 目的

本書は、CANSLIM分析基盤において、仕様書作成、仕様統合、設計レビュー、Codex実装指示、別チャット移行時の共通テンプレート運用、残検討論点チャット運用、および仕様書種別別テンプレート運用に用いる固定前提を定義する。

本書は、実装前提および文書作成運用の共通ガイドであり、設計レビュー用メモ、未検討事項管理表、個別仕様書本文、実装コードの代替ではない。

本書に未記載の事項を独自補完して、仕様本文または実装へ固定してはならない。  
未確定事項、保留事項、将来拡張、運用未決事項、詳細未決論点は未検討事項管理表を参照すること。  
未検討事項管理表は管理用であり、確定仕様の代替として扱ってはならない。

---

## 2. 文書の位置付け

本書は、以下の共通ガイドとする。

- 実装時の固定前提
- 実装禁止事項
- 必須参照先
- 補完禁止ルール
- 仕様書本文作成時の共通運用

- 仕様書種別別テンプレート
- 別チャット移行時の共通テンプレート
- 残検討論点チャット用テンプレート
- phase別オーバーレイ運用
- 回答形式
- 参照ルール
- 版管理ルール
- 仕様書本文に版番号を書かない運用
- 要約禁止の前提

本書は、個別仕様の本文、閾値本体、判定式本体、quality gate 詳細、CANSLIM各分析の詳細、Tier詳細、chart詳細、運用詳細、backtest設計を定義しない。  
必要時は役割別仕様書または未検討事項管理表を参照すること。

## 3. 基本原則

### 3.1 確定事項の扱い

- 仕様書に明記された内容のみを確定扱いとする
- 仕様書に明記されていない事項を独断で補完しない
- 既存仕様と矛盾する補完を行わない
- 仕様書には確定事項のみを記載する前提で扱う
- 未確定事項、保留事項、将来拡張、仮置き項目は仕様書本文へ戻さない
- 未確定事項は未検討事項管理表のみを受け皿とする

### 3.2 責務分離

- 取得仕様、正規化仕様、統合仕様、品質仕様、analysis仕様、集約仕様、例外仕様、screen仕様、運用仕様の責務を混同しない
- source固有の解釈を normalized 共通仕様へ持ち込まない
- role が異なる仕様書の本文を混在させない
- 個別分析仕様で上位共通前提を書き換えない
- analysis側 selector を integrated 本体競合解決の代替にしてはならない
- 集約仕様で各analysis本体条件を再定義してはならない
- 例外仕様で通常系仕様を吸い込んではならない

### 3.3 要約禁止

- 集約は要約ではなく再編とする
- 圧縮を目的に確定事項を削ってはならない
- 勝手な省略をしてはならない
- 役割が重複する記述は整理してよいが、片方にしかない確定事項は必ず残す
- 検討した論点を無記録で落としてはならない
- 本文へ採用しない論点も、未検討事項管理表送付または不採用理由のいずれかで回収する

### 3.4 本文非記載原則

以下は本書で確定しない。必要時は役割別仕様書または未検討事項管理表を参照すること。

- 取得頻度
- 日中 / 夜間の取得タイミング
- 差分判定キーの最終仕様
- 再取得頻度
- 過去何年分を正式対象とするか
- 再試行回数
- リトライ間隔
- 打切条件
- 手動確認移行条件
- 初版 vs 訂正版の最終採用優先ロジック
- 同期間複数docの採用順
- disclosure_datetime の最終利用方法
- quality_reason_code の詳細体系全文
- quality gate 閾値
- CANSLIM各分析の最終算出式・閾値
- Tier構造本体
- period exception 詳細本体
- chart判定詳細
- backtest設計
- 通知・運用監査・外部連携の詳細

## 4. 情報源参照ルール

### 4.1 参照原則

- 仕様判断は、その時点で参照可能な情報源ファイルを基に行う
- `source_index_master.md` は、情報源の常設25枠、都度参照、非格納ローカル管理を切り分ける運用索引として扱う
- プロンプト生成時は、ユーザーが個別に添付指示を書かなくても、`source_index_master.md` を起点に「参照必須ファイル」「現チャットで再添付不要な常設ファイル」「次チャット / 実行系へ実際に添付するファイル」を自動判定する
- 常設ファイルは、情報源に存在する限り、現チャットでの再アップロード対象にしないことを原則とする
- ファイル名の版番号は参照判断の主基準に使用しない
- 同一役割の仕様書が複数存在する場合は、役割整合性と内容整合性を最優先で採用判断する
- 複数候補がある場合は、採用理由と不採用理由を明示する
- 情報源ファイルに矛盾がある場合は、矛盾箇所、暫定採用案、影響範囲を示す
- 仕様書に明記された内容のみを確定扱いし、それ以外は仮定または未確定として扱う
- 未確定事項、保留事項、将来拡張、仮置き項目は仕様書本文へ戻さず、未検討事項管理表のみを受け皿とする

### 4.2 最新版判定の優先順

最新版判定は以下の優先順で行う。

1. 役割整合性
2. 内容整合性
3. 記述の新しさ
4. 更新日時

版番号は管理用識別子として扱い、参照優先順位の主基準には使用しない。

### 4.3 旧重複ファイルの扱い

- 旧再編前ファイル、旧版ファイル、重複アップロードされた旧管理表は、現行SSOTと同格参照してはならない
- 現行SSOTが存在する場合、旧ファイルは監査用参考にとどめる
- 旧ファイルにのみ残っている反映先、章立て、役割名、運用前提を、現行仕様へ戻してはならない
- 旧ファイル参照が必要な場合は、旧ファイル由来であることを明示し、本文確定根拠には使わない

### 4.4 情報源の役割認識

情報源の役割は少なくとも以下を前提とする。

- `source_index_master.md`（情報源の常設25枠・都度参照・非格納ローカル管理の索引）
- 全体設計書
- 取得仕様書
- 用語定義書
- 入力正規化・統合前処理SSOT
- 品質仕様書
- screen / runtime運用仕様書
- 実装前提ガイド
- 仕様書種別別テンプレート
- 未検討事項管理表
- 残検討フローチャート / 進捗補助資料
- 進捗まとめ文書

### 4.5 運用上の参照優先順位

運用上の参照優先順位は以下とする。

1. `source_index_master.md`
2. 現行の確定仕様書
3. 全体設計書、取得仕様書、用語定義書、入力正規化・統合前処理SSOT、品質仕様書、screen / runtime運用仕様書
4. 現行の実装前提ガイドおよび仕様書種別別テンプレート
5. 未検討事項管理表
6. 残検討フローチャート、進捗CSVなどの補助資料
7. 進捗まとめ文書
8. 旧重複ファイル

未検討事項管理表は管理用であり、確定仕様の代替として扱ってはならない。  
未検討事項管理表は、未確定事項、保留事項、将来拡張、仮置き項目の唯一の受け皿として扱う。  
残検討フローチャート、進捗CSVなどの補助資料は、phase順序や論点配置の可視化補助としてのみ扱い、仕様確定根拠へ格上げしてはならない。  
補助資料と未検討事項管理表または現行SSOTが不一致の場合は、差異を明示したうえで現行SSOTと未検討事項管理表を優先する。  
進捗まとめ文書は補助参照とし、最新仕様書より優先しない。  
旧重複ファイルは、現行SSOTが存在する場合は採用根拠に使ってはならない。

本書単独で実装判断を完結させてはならない。

## 6. 実装時の固定前提

### 6.1 統合キー

統合キーは以下で固定する。

- `stock_code`
- `fiscal_year`
- `fiscal_period`
- `statement_type`

固定式：

`stock_code + fiscal_year + fiscal_period + statement_type`

以下は統合キーではなく、補助列または監査列として扱う。

補助列：
- `disclosure_datetime`

監査列：
- `source_type`
- `source_doc_id`
- `is_revision`

`source_doc_id` を統合キーに追加してはならない。

### 6.2 fiscal_period

`fiscal_period` の許容値は以下のみとする。

- `Q1`
- `Q2`
- `Q3`
- `FY`

正規化ルール：
- `H1` は `Q2` に変換する

### 6.3 fiscal_year

`fiscal_year` は、対象決算期の期末日が属する年とする。

例：
- 2025年3月期 → `2025`

### 6.4 statement_type

`statement_type` の許容値は以下のみとする。

- `earnings_summary`
- `securities_report`
- `semiannual_report`
- `jpx_bootstrap`

source別の基本対応は以下とする。

- TDnet決算短信 → `earnings_summary`
- EDINET有価証券報告書 → `securities_report`
- EDINET半期報告書 → `semiannual_report`
- JPX bootstrap → `jpx_bootstrap`

未確定の例外変換は本書で補完しない。

### 6.5 consolidation

`consolidation` の許容値は以下のみとする。

- `consolidated`
- `non_consolidated`

採用順位は以下で固定する。

1. `consolidated`
2. 連結値が存在しない場合のみ `non_consolidated`

### 6.6 数値単位

保存単位は以下で固定する。

- 売上高：百万円
- 営業利益：百万円
- 経常利益：百万円
- 当期利益：百万円
- EPS：円

単位不明のまま保存してはならない。  
単位混在のまま保存してはならない。

### 6.7 source役割

#### TDnet

- 役割：主系
- 用途：決算短信取得
- 位置付け：最優先ソース

#### EDINET

- 役割：主系補完
- 用途：有価証券報告書、半期報告書取得
- 位置付け：TDnet補完ソース

#### JPX

- 役割：補完系
- 用途：bootstrap用途
- 位置付け：暫定ソース
- 正式値ソースとして扱わない

source優先順位は以下で固定する。

1. TDnet
2. EDINET
3. JPX

### 6.8 取得対象境界

#### TDnet

正式取得対象は以下のみ。

- 通期決算短信
- 第1四半期決算短信
- 第2四半期決算短信
- 第3四半期決算短信
- 上記の訂正版

以下は取得してもよいが、正式統合値には使わない。

- 決算補足説明資料
- 決算説明資料
- プレゼン資料
- 参考資料

以下は正式取得対象外。

- 業績予想修正
- 配当修正
- 月次売上
- KPI速報
- 自己株式取得
- 株式分割
- M&A
- 訴訟
- 一般適時開示
- 事業説明資料
- 中期経営計画

#### EDINET

正式取得対象は以下のみ。

- 有価証券報告書
- 半期報告書
- 上記の訂正提出

以下は正式取得対象外。

- 四半期報告書
- 臨時報告書
- 内部統制報告書
- 大量保有報告書
- 監査報告書単体
- 補助添付資料
- PDF中心の構造化不能資料

XBRL採用対象は、正式取得対象書類のXBRL本文のみとする。

#### JPX bootstrap

対象は以下とする。

- snapshot
- timeseries

採用条件は以下で固定する。

- `status = "OK"` のみ採用対象

固定付与値は以下とする。

- `source_type = "jpx_bootstrap"`
- `statement_type = "jpx_bootstrap"`
- `quality_status = "partial"`
- `needs_replacement = true`

同一期間に TDnet または EDINET の該当データが存在する場合、JPX bootstrap は不採用とする。

### 6.9 revision・監査列

#### is_revision

- TDnetの訂正短信は `true`
- EDINETの訂正提出は `true`
- 初版は `false`
- 判定不能時のみ `null` を許容する
- 判定不能を `false` で固定してはならない

#### disclosure_datetime

- sourceが公表した開示日時として保持する
- JSTで扱う
- 同一キー競合時の補助比較、差分取得補助、revision時系列監査に用いる

#### source_doc_id

- source内で文書を一意識別する監査用IDとして扱う
- 統合キーに使用してはならない
- `source_doc_id` 単独で正式値採用順位を決めてはならない
- 書類単位で扱い、添付ファイル単位へ分解しない

### 6.10 quality_status

`quality_status` の許容値は以下のみとする。

- `OK`
- `partial`
- `blocked`

意味は以下とする。

- `OK`：正式採用可能
- `partial`：採用可だが暫定または軽微問題あり
- `blocked`：不採用

固定前提：

- JPX bootstrap は `partial`
- `blocked` は正式採用対象外
- `partial` を `OK` に自動昇格してはならない

詳細発火条件、reason体系、quality gate 閾値は本書で補完しない。

### 6.11 レイヤ責務の固定前提

- screen layer は `integrated_fundamentals latest`、銘柄マスタ latest、price系 latest を接続し、後続分析列、chart、Tier、最終screen生成を担う
- raw layer では共通列への改名、値変換、統合判断を行ってはならない
- normalized layer は source固有表記を共通列へ変換し、後続統合可能な形へ整備する
- integrated layer は統合キー単位の競合解決を担う

### 6.12 補完禁止の実装原則

- row integrity を優先する
- column-level fallback を禁止する
- 別source継ぎ足しを禁止する
- 1行内の値混成を禁止する
- direct mapping を原則とする
- 再計算補完を独断で導入してはならない
- raw price 系の最古日を `listing_date` の代替SSOT にしてはならない
- future-dated row を current effective として先置きしてはならない
- weaker lane fallback を current effective 代替に使ってはならない

## 7. 出力前提

出力は以下を前提とする。

- Parquet形式で保存する
- `by_run_id` と `latest` を持つ構成を前提とする

対象は少なくとも以下を含む前提とする。

- `normalized_fundamentals`
- `integrated_fundamentals`
- 必要な品質・監査情報付きデータ

latest更新条件の詳細は本書で独断補完しない。

## 8. 実装禁止事項

### 8.1 列設計

- 勝手な列追加
- 既存列の型変更
- source固有列を normalized 共通列へ混在
- `source_doc_id` を統合キーに追加
- raw 原本を直接改名して normalized 代替とすること

### 8.2 閾値・判定

- 未確定閾値の独断補完
- `is_revision` 判定不能を `false` 固定
- `partial` を `OK` に自動昇格
- 未確定事項を TODO や stub のまま本実装へ固定
- quality gate 詳細を本書だけで確定扱いすること

### 8.3 欠損・数値

- 欠損のゼロ埋め
- “―” のゼロ解釈
- 単位不明値の保存
- 単位混在のまま保存
- `invalid_numeric` を `reported_null` へ勝手に変換
- 負値または 0 を自動的に invalid 扱いすること
- out-of-scope null に対して独自 reason を追加すること

### 8.4 統合

- TDnet / EDINET / JPX の優先順位改変
- JPXを正式値扱いに変更
- Q2でEDINETをTDnetより上位採用
- revisionルールの独断変更
- global な `statement_type` 最終競合順を、責務外文書で定義すること
- stronger lane 出現後に weaker lane を active fallback として採用すること
- pending replacement を main latest に先置きすること

### 8.5 運用

- latest の無条件上書き
- schema異常時の継続実行
- raw原本の自動削除
- run失敗時の無通知継続を前提にした実装
- `run_manifest` を doc粒度採用監査の主保存先にすること

### 8.6 文書運用

- 仕様書本文に版番号を書くこと
- 保存用ファイル名を本文へ流用すること
- 役割名で参照すべき箇所に実ファイル名を濫用すること
- 未確定事項を仕様書本文へ戻すこと
- 個別仕様の本文、閾値本体、判定式本体、実装コードを本書へ混在させること
- 仕様書種別が未確定のまま本文作成へ進めること
- analysis系テンプレートを集約/例外系仕様へ流用すること
- 集約/例外系テンプレートをanalysis本体へ流用すること

## 9. 仕様書本文の作成運用

本章は、仕様書本文の新規作成または改版を行う `chat_stage = reflection` のチャットにのみ適用する。  
`chat_stage = brainstorm` または `decision` の残検討論点チャットでは、本章を根拠に本文起草へ進めてはならない。  
残検討論点チャットの開始条件と段階運用は、第13章を優先する。

### 9.1 仕様書本文に版番号を書かない

- 仕様書、管理表、プロンプトの本文では、実ファイル名・版番号・v表記を記載しない
- 保存用ファイル名を提示する場合は本文外でのみ示す
- 本文中で参照が必要な場合は役割名で記載する
- 本文と保存名は分離して扱う

### 9.2 役割名ベース参照

本文中で参照が必要な場合は、以下を原則とする。

- 全体設計書
- 取得仕様書
- 用語定義書
- 入力正規化・統合前処理SSOT
- 品質仕様書
- 未検討事項管理表
- 各analysis仕様書
- chart仕様書
- Tier仕様書
- period exception 仕様書
- screen / runtime運用仕様書

### 9.3 新規作成と改版

- 上位設計または管理文書で反映先ファイル名が確定済みの場合は、そのファイル名を固定採用する
- 反映先ファイル名が確定済みの場合、別名提案をしてはならない
- 反映先ファイル名が未確定の場合のみ、保存用ファイル名を本文外で提案してよい
- 既存仕様書の修正提案時は、差分内容を明示する
- 変更提案時は、破壊的変更か、軽微修正か、後方互換ありかを明示する
- 確定仕様は勝手に変更しない
- 設計整合性または運用上の重大問題がある場合のみ、変更必要性を指摘する

### 9.4 仕様書作業開始時の固定事項

仕様書本文の新規作成または改版を開始する前に、少なくとも以下を固定すること。

- 対象仕様書名
- 反映先ファイル名
- 新規作成か既存改版か
- 仕様書種別
- 今回の対象範囲
- 非対象
- 依存仕様書
- 上位固定前提
- 責務境界
- 委譲先
- 今回確定すること
- 今回確定しないこと
- 起点論点ID
- 反映対象の論点ID一覧
- 本文へ入れる確定事項と、未検討事項管理表へ残す事項の分離
- 未確定事項の受け皿が未検討事項管理表であること

### 9.5 仕様書種別別テンプレートの使用ルール

- 仕様書本文の作成は、共通テンプレートだけで開始してはならない
- analysis系仕様書は、本書内の「analysis系仕様書テンプレート」を使用する
- 集約/例外系仕様書は、本書内の「集約/例外系仕様書テンプレート」を使用する
- 仕様書種別が未確定のまま本文作成へ進めてはならない
- analysis系テンプレートを Tier や period exception に流用してはならない
- 集約/例外系テンプレートを analysis 本体へ流用してはならない

### 9.6 「仕様書作成して」で着手できる条件

以下が固定済みの `chat_stage = reflection` チャットでは、ユーザーが「仕様書作成して」とのみ指示した場合でも着手してよい。

- 起点論点ID
- 反映対象の論点ID一覧
- 本文へ入れる確定事項と、未検討事項管理表へ残す事項の分離
- 未確定事項の受け皿

上記が未固定のチャット、または `chat_stage = brainstorm / decision` のチャットでは、本文作成へ進めてはならない。

## 10. 回答形式

原則として以下の順で出力する。

1. 指示・実行内容
2. 成果物
3. 確認事項
4. 必要説明
5. 結論

- 単純な依頼では簡略化してよい
- 確認事項は本当に必要な場合のみ最大3件まで
- ユーザーが「コピペ用」「そのまま貼付」「プレーンテキストで」と指示した場合は、説明を削り、完成文のみをプレーンテキストで提示する
- 冗長な前置き、過度な装飾、カード形式、不要な囲みは使わない
- レビュー時は、問題点、根拠、影響、修正案をセットで示す

---

## 11. 検討事項への回答ルール

- 検討事項は最初に全件まとめて列挙する
- 各項目では必要に応じて以下を示す

  - 論点
  - 推奨案
  - メリット
  - デメリット
  - 判断軸
  - 未確定なら確定に必要な論点

- 未検討事項が多い場合も、優先順位を付けたうえで一括提示を優先する
- 未検討事項抽出では設計確定や実装に進まず、論点網羅を優先する
- `chat_stage = brainstorm` では本文起草、差分本文、ファイル改版文の起草へ進めてはならない
- `chat_stage = decision` では論点確定、反映先整理、未検討事項管理表更新要否整理までとし、本文起草へ進めてはならない
- `chat_stage = reflection` かつユーザーが本文起草または差分提示を明示指示した場合のみ、仕様書本文または差分案の作成へ進んでよい
- 設計確定では、確定できる粒度まで踏み込む
- 事実と推測を分け、推測は「仮定」と明記する
- 重要前提が不足しない限り、不要な確認質問はしない
- 初回結論後は、必ず自己監査を行い、漏れ・過不足・責務衝突・旧参照残存の有無を再点検する
- 初回検討で出た重要論点を、最終出力で省略してはならない

判断軸は以下のいずれかを明示する。

- オニール準拠
- 実用性重視
- 折衷

## 12. 実装・コード関連ルール

- コードを出す場合は、Codexで実装しやすいように、仕様、入出力、例外、フォルダ構成、ファイル名を明示する
- ユーザーが入出力先フォルダを指定している場合は、その指定を優先する
- 指定がない場合のみ、最適な保存先、サブフォルダ、ファイル名を提案する
- 既存構成に影響する場合は、変更影響を明記する
- 仕様未確定のままコードへ進めない
- 実装禁止のチャットではコードを出さない
- 既存仕様へ影響する設計変更を提案する場合は、改版対象の役割別仕様書を明示する
- 実装に必要な追加判断が生じた場合は、どの仕様書のどの論点が不足しているかを明示する
- 本書の更新のみで詳細仕様を代替してはならない

## 13. 別チャット移行時のテンプレート運用

別チャットへ移行する場合は、少なくとも以下を個別指示として与えること。  
ただし、この章のテンプレートはチャット開始条件と段階制御の固定用であり、仕様書本文の直接テンプレート代替として使ってはならない。  
仕様書本文の作成または改版を行う場合は、第9章の開始条件を満たしたうえで、第16章の仕様書種別別テンプレートを必ず使用する。

### 13.1 適用区分

別チャットテンプレートは、少なくとも以下の区分を明示する。

- `chat_stage = brainstorm`：論点洗い出し、検討観点整理、依存関係確認
- `chat_stage = decision`：論点確定、推奨案決定、反映先整理、未検討事項管理表更新要否整理
- `chat_stage = reflection`：既に確定した論点を、対応する仕様書へ反映する準備または差分提示

`chat_stage = brainstorm / decision` では、本文起草、差分本文、既存仕様書の改版本文起草へ進めてはならない。  
`chat_stage = reflection` でも、ユーザーが本文起草または差分提示を明示指示するまでは、反映計画整理までにとどめる。

### 13.2 残検討論点チャットの共通コア

残検討論点チャットでは、少なくとも以下を固定する。

- 今回の目的
- `target_issue_id`
- `target_phase`
- `chat_stage`
- `source_index_master.md`
- 論点の現在状態
- 関連論点ID（参照のみ。別論点の確定はしない）
- 反映候補または `planned_reflection_specs`
- 今回扱う範囲
- 今回扱わない範囲
- 禁止事項
- 添付補助資料の有無
- 出力要件

対象は原則1論点IDとし、別論点へ拡張して確定してはならない。  
関連論点を扱う場合も、影響列挙、依存関係確認、反映先整合確認までにとどめる。

### 13.3 情報源の扱い

- 仕様判断は参照可能な情報源ファイルに基づくこと
- `source_index_master.md` を起点に常設25枠と都度参照対象を確認すること
- 別チャット投入用プロンプトを生成する場合は、`source_index_master.md` を基に、参照必須ファイル、現チャットで再添付不要な常設ファイル、次チャットへ実際に添付するファイルを分離して示すこと
- 常設ファイルは、情報源に存在する限り、現チャットでの再アップロード対象にしないことを原則とする
- 版番号ではなく役割整合性と内容整合性で採用判断すること
- 明記された内容のみを確定扱いすること
- 未確定事項は未検討事項管理表へ集約すること
- 旧重複ファイルは現行SSOTと同格参照しないこと

残検討論点チャットでは、参照優先順位を少なくとも以下で固定する。

1. `source_index_master.md`
2. 関連する現行確定仕様書
3. 全体設計書
4. 本ガイドおよび仕様書種別別テンプレート
5. 未検討事項管理表
6. 残検討フローチャート、進捗CSVなどの添付補助資料
7. 旧重複ファイル

未検討事項管理表は未確定事項の唯一の受け皿であり、確定仕様の代替として扱ってはならない。  
残検討フローチャート、進捗CSVなどの補助資料は、phase順序、論点配置、進捗状態の可視化補助としてのみ扱い、仕様確定根拠に格上げしてはならない。  
補助資料と未検討事項管理表または現行SSOTが不一致の場合は、差異を明示したうえで、現行SSOTと未検討事項管理表を優先する。

### 13.4 作業開始時の固定事項

残検討論点チャットでは、少なくとも以下を固定する。

- `target_issue_id`
- `target_phase`
- `chat_stage`
- 論点の現在状態
- 依存仕様書一覧
- 上位固定前提
- 責務境界
- 委譲先
- 今回確定すること
- 今回確定しないこと
- 未検討事項管理表へ残す事項
- 出力要件

`chat_stage = reflection` の場合は、追加で以下を固定する。

- 対象仕様書名
- 反映先ファイル名
- 仕様書種別
- 新規作成か既存改版か
- 起点論点ID
- 反映対象の論点ID一覧
- 本文へ入れる確定事項と、未検討事項管理表へ残す事項の分離

### 13.5 成果物指定

`chat_stage` ごとの成果物指定は、少なくとも以下を前提とする。

#### brainstorm

- 論点一覧
- 依存関係
- 参照不足
- 責務衝突
- 別論点へ送る必要がある項目
- 今後の確定順序

#### decision

- 推奨案
- 問題点、根拠、影響、修正方針
- 確定内容
- 未確定として残す内容
- 反映先仕様書
- 未検討事項管理表更新要否
- 次段で反映すべき事項

#### reflection

- 反映対象整理
- 差分案
- 旧参照残存の有無
- 反映先ごとの修正要否
- 未反映として残す論点
- 進捗まとめ文書用のコピペ文

`chat_stage = brainstorm / decision` で「完成本文全文」を既定成果物にしてはならない。  
`chat_stage = reflection` でも、ユーザーの明示指示があるまでは本文起草を既定成果物にしてはならない。

### 13.6 phase別オーバーレイ

テンプレートは、共通コア1本に phase別オーバーレイを重ねる方式を原則とする。  
phaseごとに全文テンプレートを分岐させるのではなく、以下の重点だけを差し替える。

- Phase0 管理是正：文書統治、参照優先順位、状態管理、反映ルール、責務境界の補正を重点確認する
- Phase1 正規化確定：用語、列契約、正規化、初期gate、入力境界、品質接続を重点確認する
- Phase2 runtime境界：取得仕様、runtime、latest更新条件、run_manifest、障害運用境界を重点確認する
- Phase3 exposure確定：出力接続、公開列、screen接続、下流露出境界を重点確認する
- Phase4 補助指標：補助指標の役割、入力契約、analysisやscreenとの責務境界を重点確認する
- Phase5 拡張群：将来拡張、例外、任意採用候補、補助ルールの早期本文流入防止を重点確認する

共通コアに載らない phase固有の参照先や禁止事項のみ、オーバーレイとして追加する。

### 13.7 フローチャート・進捗ファイルの扱い

残検討論点チャットでは、必要に応じて残検討フローチャートや進捗CSVを添付してよい。  
ただし、これらは補助資料であり、以下を守る。

- 添付がない場合でも、現行SSOTと未検討事項管理表を基に検討を進める
- 添付がある場合も、仕様確定根拠の代替に使わない
- node順序、phase順序、進捗状態の可視化補助として扱う
- 論点ID、状態、反映先が未検討事項管理表と不一致の場合は、不一致自体を指摘する
- 補助資料の記載を理由に、確定仕様や未確定状態を上書きしてはならない

### 13.8 本文起草開始条件

以下が固定済みの `chat_stage = reflection` チャットでは、ユーザーが「仕様書作成して」とだけ指示した場合でも、追加前置き最小で着手してよい。

- 対象仕様書名

- 新規作成か既存改版か

- 未確定事項の受け皿

これらが未固定のチャット、または `chat_stage = brainstorm / decision` のチャットでは、本文作成へ進めてはならない。

### 13.9 進捗まとめ前提

ファイルへの書き込みはユーザー側が行う前提とし、こちらはコピペ用本文を提示する。

## 17. Codex投入時の入力不足防止ルール

### 17.1 適用目的

本章は、Codexへ実装、修正、レビュー、テスト修正を依頼する際に、入力不足による曖昧実装、仕様逸脱、参照漏れを防ぐための固定運用を定義する。

本章は、プロンプト生成チャットと Codex 実行チャットを分けて扱う。  
プロンプト生成チャットでは、`source_index_master.md` に常設済みのファイルを再アップロードする前提を置かない。  
一方で、Codex 実行側が当該情報源へアクセスできない場合は、Codex へ渡す情報をプロンプト本文と実際に添付したファイルのみで成立させるものとして扱う。

### 17.2 Codexへ渡す入力の最低構成

Codexへ依頼する場合は、少なくとも以下を明示すること。

- 今回の目的
- 今回の作業範囲
- 今回の作業範囲外
- 変更対象ファイル
- 参照必須ファイル
- 現チャットで再添付不要な常設ファイル
- 次チャット / 実行系へ実際に添付するファイル
- 実装禁止事項
- 受入条件
- 未確定事項の返却先
- 出力形式

上記が不足している場合、Codexへ「完成コード生成」を依頼してはならない。  
不足がある場合は、設計レビュー、差分案、参照不足指摘、追加添付ファイル特定までにとどめる。

### 17.2A Codex投入時の推論労力明示ルール

Codex用プロンプトを生成する場合は、添付ファイル名一覧の提示に加えて、推奨する `reasoning_effort` を必ず明示すること。

Codex用プロンプト本文には、少なくとも以下を含める。

- 推奨 `reasoning_effort`
- 当該推奨理由の簡潔な説明
- 参照必須ファイル
- 現チャットで再添付不要な常設ファイル
- 次チャット / 実行系へ実際に添付するファイル

`reasoning_effort` の推奨値は、作業の性質に応じて以下を原則とする。

- `low`：軽微修正、文言修正、rename、局所的テスト修正
- `medium`：通常の実装、既存仕様のコード化、既存仕様に従う機能追加
- `high`：複数仕様書横断の整合確認が必要な実装、責務境界の衝突確認、不整合調査、抽象設計を伴う作業

Phase 3 の通常実装では、原則として `medium` を推奨する。  
ただし、複数仕様書横断の衝突確認、period exception、競合解決、複雑な chart / trade rule 周辺など、仕様解釈負荷が高い場合のみ `high` を推奨してよい。

推奨値を明示せずに Codex用プロンプトを完成扱いにしてはならない。

### 17.3 入力不足時に進めてよい条件

以下を満たす場合に限り、Codexへ実装依頼を行ってよい。

- `source_index_master.md` により、今回参照すべき常設 / 都度参照ファイルが特定できる
- ChatGPT が、ユーザーの個別指示がなくても、参照必須 / 再添付不要 / 実添付の三区分を自動判定できる
- 変更対象ファイルが明示されている
- 変更対象に対応する役割別仕様書が添付または常設参照で確定できる
- 受入条件が文章で明示されている
- 未確定事項を本文へ戻さないことが明示されている

### 17.4 入力不足時に進めてはならない条件

以下に該当する場合、Codexへ完成コード生成を依頼してはならない。

- 変更対象ファイルが未指定
- 対応する役割別仕様書が未特定
- 添付ファイルが不足しており、仕様本文確認ができない
- 受入条件がない
- 未確定事項の扱いが未指定
- 参照必須ファイル、現チャットで再添付不要な常設ファイル、実際の添付ファイルが分離されていない
- `source_index_master.md` を読まずに、ユーザーが添付要否を別途判断する前提にしている
- 「前チャットの流れを読んで判断する」ことを前提にしている

この場合は、まず不足ファイル名一覧、添付優先順位、不足時の影響を整理してから次段へ進める。

### 17.5 Codex実装指示テンプレート

Codexへ渡す本文は、少なくとも以下の型を満たすこと。

### 17.6 Codex出力の受入確認ルール

Codexの出力を受けた後は、少なくとも以下を確認すること。

- 変更対象外ファイルへ不要変更が波及していないか
- 参照必須ファイルの責務境界を壊していないか
- 受入条件を満たしているか
- テスト結果が示されているか
- 未確定事項を勝手に本文やコードへ固定していないか
- 仮定で進めた箇所が明示されているか
- 既存コードとの整合性が壊れていないか

### 17.7 添付ファイル指示の原則

Codex 投入時または次チャット投入用のプロンプト生成時は、少なくとも以下の 3 区分を分けて記載すること。  

1. `required_reference_files`  
2. `no_reupload_if_source_resident`  
3. `attached_files`  

`source_index_master.md` に常設と定義され、かつ現チャットの情報源に存在するファイルは、原則として `no_reupload_if_source_resident` に置く。  
ユーザーが個別に「これは添付不要」と指示しなくても、ChatGPT が本ルールを読んで自動判定する前提で運用する。  

ただし、以下のいずれかに該当する場合は、常設ファイルであっても `attached_files` に含めてよい。  

- 次チャット / 実行系が当該情報源へアクセスできない  
- 情報源へまだ反映されていない新版を使う必要がある  
- 現行の情報源本文だけでは不足し、当該版本文を明示共有する必要がある  

参照必須ファイルに書いただけで、対象実行系が自動参照できる前提を置いてはならない。  
一方で、現チャットの情報源に存在する常設ファイルを、理由なく毎回再アップロードする前提も置いてはならない。  
添付していないファイルは、対象実行系が当該情報源を読めない場合、参照名のみでは未読扱いとなる前提で入力を組むこと。
