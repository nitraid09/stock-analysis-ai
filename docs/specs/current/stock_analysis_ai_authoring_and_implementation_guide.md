# 株式分析AI authoring / implementation guide

- 文書名: 株式分析AI authoring / implementation guide
- 文書ID: stock_analysis_ai_authoring_and_implementation_guide
- 版: v0.1
- 作成日: 2026-04-19
- 位置づけ: 株式分析AIプロジェクトにおける仕様書作成、仕様反映、設計レビュー、Codex実装指示、別チャット移行時のテンプレート運用に用いる共通ガイド

---

## 1. 目的

本書は、株式分析AIプロジェクトにおいて、以下の作業を一貫した前提で進めるための共通ガイドを定義する。

- 仕様書の新規作成および改版
- 残課題管理表の更新
- decision / reflection 用レビュー
- Codex 実装指示プロンプトの作成
- acceptance decision memo、handoff、進捗まとめ文書の作成
- 別チャット移行時の開始条件固定

本書は、文書作成運用と実装指示運用の共通前提を定めるものであり、プロジェクト本文仕様、残課題管理表、個別 acceptance memo、個別 Codex handoff、実装コードの代替ではない。

本書に未記載の事項を独自補完して、仕様本文または実装へ固定してはならない。未確定事項、保留事項、将来拡張、技術方式未決事項は、本文へ戻さず残課題管理表で管理する。

---

## 2. 文書の位置付け

本書は、少なくとも以下の共通ガイドとして扱う。

- 参照優先順位
- project 固定前提
- prompt 作成時の禁止事項
- 仕様反映開始条件
- review / decision / reflection / implementation の段階運用
- 文書種別ごとの最低記載要件
- Codex handoff 作成時の固定観点
- 旧版・別プロジェクト文書の扱い

本書は、以下を定義しない。

- 売買ルール本文の最終本体
- 正本保存基盤の completed SQL 本文
- HTML / CSS / JavaScript の最終見た目
- DB migration 手段の技術詳細
- broker / market / IR の取得実装詳細
- 個別 issue の最終結論本文
- 実装コードそのもの

必要時は、現行のプロジェクト目的・運用仕様書、残課題管理表、個別 handoff、個別 acceptance memo を参照すること。

## 3. 基本原則

### 3.1 確定事項の扱い

- 現行仕様書に明記された内容のみを確定事項として扱う
- 明記されていない事項を独断で補完しない
- 別チャットの口頭整理、過去の暫定案、旧版の残骸を本文確定根拠へ昇格させない
- 未確定事項は本文へ戻さず、残課題管理表へ残す
- 実装指示でも、未確定事項は「仮定」または「残課題」として分離する

### 3.2 責務分離

- プロジェクト本文仕様、残課題管理表、acceptance memo、Codex handoff の責務を混同しない
- project SSOT と handoff 用の作業指示文を同格文書として扱わない
- storage、orchestration、render、publish、archive の責務を混同しない
- HTML 表示責務と正本更新責務を同一層へ持ち込まない
- 表示都合を理由に正本構造を増やさない

### 3.3 要約禁止の原則

- 反映は要約ではなく再編として扱う
- 重要な確定事項を圧縮目的で落とさない
- 既存記述が長くても、境界条件、不変条件、禁止事項、例外条件は残す
- 省略した論点がある場合は、本文非対象か残課題送りかを明示する

### 3.4 別プロジェクトの文書流用原則

- 別プロジェクト文書は、構成参考には使ってよい
- ただし、用語、責務、情報源運用、反映先、段階区分、テンプレート種別を無修正で流用してはならない
- 特に `source_index_master.md` 前提、CANSLIM 固有の analysis / aggregation / exception 分類、backtest 固有責務は、本プロジェクトへ逆流させない

## 4. 情報源参照ルール

### 4.1 参照原則

仕様判断および prompt 生成は、その時点で参照可能な現行ファイルに基づいて行う。

本プロジェクトでは、`source_index_master.md` を参照前提にしない。別系プロジェクトの情報源索引運用を、本プロジェクトの仕様、反映先、実装指示へ逆流させてはならない。

### 4.2 参照優先順位

参照優先順位は、少なくとも以下を原則とする。

1. 現行の株式分析AIプロジェクト目的・運用仕様書
2. 現行の株式分析AI残課題管理表
3. 今回対象の acceptance decision memo / decision memo / reflected draft
4. 今回対象の Codex handoff / 実装指示書
5. 本ガイド
6. 進捗まとめ、作業メモ、補助ドラフト
7. 旧版ファイル
8. 別プロジェクトの文書

### 4.3 最新版判定

同一役割の候補文書が複数ある場合、採用優先順は以下とする。

1. 役割整合性
2. 内容整合性
3. 記述の新しさ
4. 更新日時

ファイル名の版番号は参考情報として扱ってよいが、主判定基準にしない。

### 4.4 旧版・別プロジェクト文書の扱い

- 現行SSOTが存在する場合、旧版は監査用参考にとどめる
- 旧版にしか残っていない章立てや表現を、自動的に現行へ戻さない
- 別プロジェクト文書は構成参考には使えても、確定根拠には使わない
- 別プロジェクト文書由来の用語を使う場合は、本プロジェクト用語へ言い換える

## 5. 本プロジェクトの固定前提

### 5.1 目的・運用前提

- プロジェクトの正式目的は 1年間のトータルリターン最大化とする
- 評価は勝率主義ではなく、期首評価額に対する期末評価額の増減率を主軸とする
- AI は売買判断支援を担うが、最終執行責任はユーザー側にある
- 初版運用は現物買いのみを対象とし、信用取引、空売り、時間外取引は対象外とする

### 5.2 記録・表示前提

- 正式な閲覧形式は HTML とする
- 正本は構造化データであり、HTML は再生成表示物として扱う
- 正本更新と HTML 再生成は責務分離する
- 正本更新が確定した後に HTML 再生成が失敗しても、正本更新までは巻き戻さない

### 5.3 保存ルートと保管責務

- 保存ルートは `C:\projects\AI分析` を前提とする
- 正本データは `master` 配下、HTML 生成物は `generated_html` 配下、月次凍結物は `archive` 配下、運用文書は `docs` 配下に置く
- `generated_html\latest` は、最後に受入条件を満たした successful generation のみを保持する
- `generated_html\generations` は generation 単位 staging / publish 履歴を保持する
- `archive\monthly` は successful generation 由来の月次凍結参照用とする

### 5.4 正本構造の固定境界

- 正本保存基盤の中核は SQLite を前提とする複合方式とする
- 6記録単位は、提案、注文・約定、保有スナップショット、振り返り、米国株ウォッチリスト・仮想売買、米国株実売買パイロットとする
- supporting table は許容してよいが、新しい独立正本、集計専用正本、トップ画面専用正本、月次専用正本を追加してはならない
- `payload_json` は正式列契約の代替として扱わない
- child 系および supporting table の更新粒度は一律の親キー単位置換にしない

### 5.5 HTML 再生成・URL 契約の固定前提

- HTML 出力単位は原則として標準画面単位とする
- proposal_detail は `proposal_id` 単位、monthly_review は月次キー単位の個別出力を許容する
- URL 責務は、path = 画面責務または自然な個別参照単位、query = 共有状態、hash = 同一ページ内 stable key ジャンプとする
- `empty`、`none`、`not_applicable`、未記録、評価対象外、参照不能、再生成失敗、エラー状態は同一表示へ潰さない

## 6. 実装・文書化の禁止事項

### 6.1 共通禁止事項

- 仕様未記載事項の独自補完
- 未確定事項の本文固定
- 別プロジェクトの前提を無修正で流用
- 旧版を現行SSOTと同格扱いすること
- silent overwrite
- 例外握りつぶし
- 受入未達のまま完了扱い

### 6.2 本プロジェクト固有の禁止事項

- `source_index_master.md` 前提の導入
- HTML テンプレートを物理 table や SQLite 列へ直結すること
- render 層へ正本更新責務、change set 解決責務、publish 判定責務を持たせること
- generation 失敗時に `generated_html\latest` を更新すること
- latest 切替失敗時に broken latest を標準状態として残すこと
- 新しい独立正本、集計専用正本、画面専用正本の追加
- child 直付け evidence を初版標準へ拡張すること
- 注文状態と照合状態を単一 generic 状態列へ潰すこと
- 非アクティブ提案、見送り提案、新規建てゼロ提案、候補外、無効化対象を画面都合で除外すること

### 6.3 文書運用上の注意

- 別プロジェクト由来の章立てをそのまま採用しない
- 本プロジェクトで未採用のテンプレート分類を固定しない
- 版番号運用は本プロジェクトの既存文書実態に合わせる。別プロジェクトの「本文に版番号を書かない」ルールを自動継承しない

## 7. 文書種別ごとの作成運用

### 7.1 核となる文書種別

本プロジェクトでは、少なくとも以下の文書種別を区別する。

1. プロジェクト目的・運用仕様書
2. 残課題管理表
3. acceptance decision memo / decision memo
4. Codex handoff / 実装指示書
5. 進捗まとめ文書

### 7.2 プロジェクト目的・運用仕様書の更新

目的・運用仕様書へ反映する場合は、少なくとも以下を固定する。

- 対象仕様書名
- 反映先ファイル名
- 新規作成か既存改版か
- 今回の対象範囲
- 非対象
- 上位固定前提
- 起点論点ID または対象論点群
- 本文へ入れる確定事項
- 残課題管理表へ残す事項

本文へ反映してよいのは、decision 済みの確定事項のみとする。

### 7.3 残課題管理表の更新

残課題管理表更新では、少なくとも以下を明示する。

- issue_id
- phase
- status_before / status_after
- planned_reflection_specs
- management_table_action
- 決定要旨
- 本文反映後の扱い

残課題管理表は管理用であり、仕様本文の代替として扱ってはならない。

### 7.4 acceptance decision memo / decision memo

memo では、少なくとも以下を明示する。

- 何を受け入れたか、または差し戻したか
- どの仕様と整合したか
- 何が未完了か
- 何を本文へ反映すべきか
- 何を残課題へ残すか

memo は決定・受入の根拠整理であり、SSOT 本文の代替ではない。

### 7.5 Codex handoff / 実装指示書

Codex handoff では、少なくとも以下を固定する。

- 今回の目的
- reasoning_effort と理由
- 作業範囲
- 作業範囲外
- 変更対象ファイル
- 参照必須ファイル
- 実装禁止事項
- 固定事項
- 受入条件
- 未確定事項の扱い
- 出力形式

実装指示では、仕様とコード責務の境界を壊さないことを最優先とする。

## 8. チャット段階運用

### 8.1 段階区分

本プロジェクトでは、少なくとも以下の `chat_stage` を用いてよい。

- `brainstorm`: 論点洗い出し、依存関係確認、未確定事項抽出
- `decision`: 推奨案提示、確定 / 未確定整理、反映先整理
- `reflection`: 確定事項の本文反映案作成または差分化
- `implementation`: Codex handoff 作成、実装指示具体化

### 8.2 各段階でやってよいこと

#### brainstorm

- 論点一覧化
- 依存関係整理
- 責務衝突の指摘
- 残課題抽出

本文起草や実装指示確定へ進めてはならない。

#### decision

- 推奨案提示
- 問題点、根拠、影響、修正方針の提示
- 確定事項と未確定事項の分離
- 反映先仕様書または管理表の整理

未確定事項を本文へ流し込んではならない。

#### reflection

- 反映対象整理
- 差分案または全文版作成
- 旧参照残存の有無確認
- 反映後自己監査

reflection では、対象仕様書名、反映先ファイル名、対象範囲が未固定のまま開始してはならない。

#### implementation

- Codex handoff 作成
- 変更対象と禁止事項の固定
- 受入条件の明文化
- 未確定事項の残課題化

implementation では、現行仕様と矛盾する実装自由度を与えてはならない。

## 9. prompt 作成時の固定観点

### 9.1 最低記載要件

prompt 作成時は、少なくとも以下を明示する。

- 役割
- 今回の目的
- 参照必須ファイル
- 対象範囲
- 非対象
- 禁止事項
- 固定事項
- 未確定事項の扱い
- 出力形式

### 9.2 反映系 prompt の固定観点

反映系 prompt では、以下を必ず分離する。

- 本文へ入れる確定事項
- 残課題管理表へ残す事項
- 旧記述の削除対象
- 追記対象
- 変更不要箇所

### 9.3 Codex handoff の固定観点

Codex 向け prompt では、以下を曖昧にしない。

- 実装対象責務
- 実装禁止責務
- 受入条件
- テスト要求
- 変更対象外への波及禁止
- 仮定を置ける範囲

### 9.4 別プロジェクト文書を参考にする場合の変換ルール

別プロジェクト文書を参考にする場合は、最低でも以下を置換または削除する。

- プロジェクト名
- 情報源運用
- 用語体系
- 文書種別分類
- 反映先ファイル名
- 固有責務
- 非対象一覧
- 参照必須ファイル
- 実装禁止事項

---

## 10. 回答形式

原則として以下の順で出力する。

1. 指示・実行内容
2. 成果物
3. 確認事項
4. 必要説明
5. 結論

- 単純な依頼では簡略化してよい
- 確認事項は本当に必要な場合のみ最大3件まで
- レビュー時は、問題点、根拠、影響、修正案をセットで示す
- コピペ用を求められた場合は、説明を削り完成文を優先する

---

## 11. 本ガイドを使って作るべきテンプレート

本プロジェクトでは、少なくとも以下のテンプレートを本書準拠で整備してよい。

- decision review prompt テンプレート
- reflection / 文書反映 prompt テンプレート
- Codex handoff テンプレート
- acceptance decision memo テンプレート
- 残課題管理表更新テンプレート

別プロジェクトの analysis系 / aggregation系 / exception系テンプレートは、そのままの分類では採用しない。

---

## 12. 結論

本書は、株式分析AIプロジェクトにおける仕様反映、残課題更新、受入判断、Codex実装指示のための共通ガイドである。

本プロジェクトでは、別系プロジェクトの `source_index_master.md` 前提や CANSLIM 固有の文書分類を流用してはならない。現行のプロジェクト目的・運用仕様書、残課題管理表、個別 decision memo、個別 handoff を参照し、確定事項のみを本文へ反映し、未確定事項は残課題へ残すこと。
